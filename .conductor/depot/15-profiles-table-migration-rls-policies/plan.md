## Phase 1: Create migration SQL file with profiles table, indexes, constraints, and triggers

**Goal:** Create the complete migration SQL file containing the profiles table DDL, all indexes, all constraints, the auto-create trigger (with slug collision handling), the updated_at trigger, and the UPDATE trigger that prevents modification of system-managed fields.
**Verify:** `pnpm build`

### Task 1.1: Create the profiles table migration SQL file with full schema

Write the migration SQL file at `supabase/migrations/{timestamp}_create_profiles_table.sql`. The file must contain, in order:

1. `CREATE TABLE profiles (...)` with all columns from the issue schema (id UUID PK referencing auth.users, display_name, slug, avatar_url, bio, primary_species, primary_technique, home_state, years_fishing, is_seller, is_stripe_connected, stripe_account_id, stripe_onboarding_status, average_rating, review_count, total_transactions, last_seen_at, response_time_hours, onboarding_completed_at, deleted_at, notification_preferences JSONB with default, created_at, updated_at).
2. All four indexes: `profiles_display_name_unique` (unique, on LOWER(display_name)), `profiles_slug_unique` (unique, on slug), `profiles_home_state_idx`, `profiles_is_seller_idx` (partial WHERE is_seller = TRUE).
3. All constraints: `display_name_length` CHECK (char_length BETWEEN 3 AND 40), `bio_length` CHECK (NULL or <= 280), `slug_format` CHECK regex `^[a-z0-9][a-z0-9-]*[a-z0-9]$`.

**Files:** `supabase/migrations/20260319000000_create_profiles_table.sql`
**AC:** The file contains a valid CREATE TABLE statement with all 22 columns, 4 indexes, and 3 constraints matching the issue schema exactly.
**Expert Domains:** supabase

### Task 1.2: Add triggers to the migration SQL file

Append to the same migration file three trigger functions and their triggers:

1. **`handle_new_user()` trigger function** — Fires AFTER INSERT on `auth.users`. Creates a profiles row with `id = NEW.id`, `display_name` derived from `NEW.raw_user_meta_data->>'firstName'` (fallback to `'User'`), and `slug` generated by lowercasing the display_name, replacing non-alphanumeric characters with hyphens, and trimming leading/trailing hyphens. Must handle slug collisions by appending a random 4-digit numeric suffix, retrying up to 5 times. On collision exhaustion, raise an exception.
2. **`handle_updated_at()` trigger function** — Fires BEFORE UPDATE on `profiles`. Sets `NEW.updated_at = NOW()`.
3. **`handle_profiles_system_fields()` trigger function** — Fires BEFORE UPDATE on `profiles`. Prevents user modification of system-managed fields by resetting `NEW.stripe_account_id`, `NEW.is_stripe_connected`, `NEW.stripe_onboarding_status`, `NEW.average_rating`, `NEW.review_count`, and `NEW.total_transactions` to their `OLD` values.

Each function must use `SECURITY DEFINER` where appropriate (the auto-create trigger needs it to write to profiles from an auth schema trigger). Create the corresponding `CREATE TRIGGER` statements binding each function to its table.

**Files:** `supabase/migrations/20260319000000_create_profiles_table.sql`
**AC:** Three trigger functions and three triggers exist in the migration file. The auto-create trigger handles slug collisions with up to 5 retries. The updated_at trigger fires BEFORE UPDATE. The system fields trigger resets all 6 system-managed fields to OLD values.
**Expert Domains:** supabase

## Phase 2: Add RLS policies for profiles, products, and product_images

**Goal:** Enable RLS on all three tables and create the exact policies specified in the issue and design spec Sections 7.1 and 7.2.
**Verify:** `pnpm build`

### Task 2.1: Add profiles RLS policies to the migration SQL file

Append to the migration file: `ALTER TABLE profiles ENABLE ROW LEVEL SECURITY` followed by three policies:

1. **SELECT** — `"Profiles are viewable by everyone"` — grants SELECT to `authenticated, anon` with `USING (true)`.
2. **INSERT** — `"Users can insert own profile"` — grants INSERT to `authenticated` with `WITH CHECK ((SELECT auth.uid()) = id)`.
3. **UPDATE** — `"Users can update own profile"` — grants UPDATE to `authenticated` with `USING ((SELECT auth.uid()) = id)` and `WITH CHECK ((SELECT auth.uid()) = id)`.

No DELETE policy — profiles are soft-deleted via `deleted_at`.

**Files:** `supabase/migrations/20260319000000_create_profiles_table.sql`
**AC:** RLS is enabled on profiles with exactly 3 policies (SELECT, INSERT, UPDATE). No DELETE policy exists.
**Expert Domains:** supabase

### Task 2.2: Add products and product_images RLS policies to the migration SQL file

Append to the migration file the RLS policies from the design spec. For **products** (Section 7.1): enable RLS, then create 4 policies — SELECT (everyone), INSERT (own), UPDATE (own), DELETE (own) — matching the exact SQL from the design spec. For **product_images** (Section 7.2): enable RLS, then create 4 policies — SELECT (everyone), INSERT (own products via EXISTS subquery), UPDATE (own products via EXISTS subquery), DELETE (own products via EXISTS subquery) — matching the exact SQL from the design spec.

**Files:** `supabase/migrations/20260319000000_create_profiles_table.sql`
**AC:** Products table has RLS enabled with 4 policies matching design spec Section 7.1 exactly. Product_images table has RLS enabled with 4 policies matching design spec Section 7.2 exactly.
**Expert Domains:** supabase
